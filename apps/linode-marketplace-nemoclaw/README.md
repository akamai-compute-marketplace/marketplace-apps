# Akamai NemoClaw Quick Deploy App

NemoClaw is an open-source CLI orchestrator from NVIDIA that runs the OpenClaw AI agent inside a Docker sandbox and proxies its network access through a companion process called OpenShell. It is not a typical web app: it is configured through a guided CLI wizard (`nemoclaw onboard`) that picks an inference provider, collects credentials, and starts the sandboxed dashboard.

# Post Deployment

When the NemoClaw Quick Deploy App has finished installing, you will need to log into your server to perform the onboarding. The onboarding script will kick off only when you log in as the `root` user. Here are some notes about the deployment:

- NemoClaw runs as a limited user on the instance called `nemoclaw`. The `nemoclaw` user has
  limited scope defined in `/etc/sudoers.d/nemoclaw` and is a member of the `docker` group.
- A sudo user is created on the instance for system administrative purposes.
- The onboarding script is located in `/etc/profile.d/nemoclaw_onboarding.sh`. Once onboarding is
  complete, the script automatically removes itself so it won't prompt again on next login.
- NemoClaw's OpenShell gateway runs as a `systemd --user` service (`openshell-gateway.service`)
  under the `nemoclaw` user, pre-installed by this playbook so `nemoclaw onboard` can supervise it
  with automatic restarts.

## NemoClaw Onboarding

To perform the NemoClaw onboarding you will need to log into the system as `root` (if enabled) or
the sudo user. When `root` logs in, it triggers the onboarding script
automatically.

If you're logging into the server with the sudo user:

1. Grab the sudo password from `/home/admin/.credentials`.
2. Become `root` by running `sudo su -`. When prompted for the sudo password, paste it from the
  `/home/admin/.credentials` file.
3. Answer the onboarding prompt (`y`) to run `nemoclaw onboard` as the `nemoclaw` user. You will be
   asked to choose an inference provider (a cloud API such as NVIDIA Build/Endpoints, OpenAI,
   Anthropic, OpenRouter, Gemini, or a self-hosted OpenAI-compatible server) and supply its
   credential — this cannot be pre-generated at deploy time.

## Accessing the NemoClaw Dashboard

This deployment uses SSH tunneling for dashboard access. The dashboard is not exposed on a public
HTTP(S) endpoint.

Dashboard access steps (from your local machine):

```bash
ssh -L 18789:127.0.0.1:18789 root@{{ _domain }}
sudo -i -u nemoclaw nemoclaw <sandbox-name> dashboard-url
```

Then visit `http://127.0.0.1:18789` locally.

Note: `sudo -i -u nemoclaw nemoclaw <sandbox-name> dashboard-url` will show a Dashboard URL similar to the one below:

```bash
Dashboard URL:
http://127.0.0.1:18789/#token=DbsMSK8L7eyavy2FVR4A1z6YORErn8V9jjWzP4HWL0A
Treat this URL like a password -- do not log, share, or commit it.
```

## Technical Deep Dive: Running Tasks in the Sandbox

### 1. Understanding the Sandbox Boundary

The OpenClaw agent does not run directly on the host OS. NemoClaw isolates agent execution inside a sandbox to reduce blast radius from accidental destructive commands or unauthorized host access.

```text
┌────────────────────────────────────────────────────────────────────────┐
│                        AKAMAI LINODE INSTANCE                         │
│                                                                        │
│  ┌───────────────────────┐                  ┌───────────────────────┐  │
│  │    Host OS (Ubuntu)   │                  │  Docker Sandbox       │  │
│  │                       │                  │                       │  │
│  │  • OpenShell Gateway  │ ◄─ Brokered I/O  │  • OpenClaw Agent     │  │
│  │  • NemoClaw CLI       │                  │  • Agent Tools        │  │
│  │  • Docker Daemon      │                  │  • Local Filesystem   │  │
│  └───────────────────────┘                  └───────────────────────┘  │
└────────────────────────────────────────────────────────────────────────┘
```

- Host responsibilities: runs `openshell-gateway`, NemoClaw CLI, and Docker runtime.
- Sandbox responsibilities: runs the agent and task execution context.
- Isolation behavior: destructive sandbox commands impact the container workspace, not the host root filesystem.

### 2. Next Steps: Managing Your Workspace

Files generated by the agent are written inside the sandbox workspace.

- Optional: during onboarding, configure workspace paths if you want persistent host data integration.
- Direct inspection: you can inspect sandbox files from host CLI.

```bash
# List running sandbox containers
docker ps | grep nemoclaw

# Open a shell in the running sandbox container
docker exec -it <container_id_or_name> /bin/bash
```

### 3. Configuring Agent Network Egress (OpenShell Policy)

OpenShell acts as the network policy broker for sandbox traffic.

- Sandbox egress is mediated through OpenShell.
- Default posture is restrictive.

If needed, adjust OpenShell policy/config under the `nemoclaw` user configuration (commonly under `/home/nemoclaw/.config/openshell/`).

### 4. Selecting and Benchmarking Models

After onboarding, you can run task comparisons across configured providers.

- Use lightweight tasks first to validate latency, tool-calling quality, and determinism.
- For heavier implementation tasks, test at least two models against the same prompt and compare outputs.
- Add explicit guardrails in your system prompt to cap runaway loops.

Example guardrail:

```text
If you cannot solve the task within 10 steps, stop and output a concise report of attempted approaches and blockers.
```

### 5. Cleaning Up and Resetting Sandboxes

Periodic sandbox resets help clear stale state and temporary package drift.

```bash
sudo -i -u nemoclaw nemoclaw <sandbox-name> destroy
sudo -i -u nemoclaw nemoclaw <sandbox-name> create
```

This recreates the sandbox runtime environment while preserving your external provider/account setup flow.
